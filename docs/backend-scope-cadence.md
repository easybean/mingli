# 后端阶段设计备忘：多尺度关卡的持久化与刷新机制

> 这是**后端改造阶段**才落地的设计，当前纯前端阶段不做（见 CLAUDE.md 阶段约束）。
> 记录于 2026-06-19 的一次讨论：纯前端版选择只在内存、刷新即丢；做后端要记录用户、且
> 今日/月/年/十年/一生这几种关卡不可能"一年做一次"，需要一套刷新/持久化机制。

## 核心判断：五个尺度不能用同一种"刷 N 题"机制

五个尺度的自然频率差几个数量级（日/月/年/十年/一生）。不能都做成"play through N 张卡"。分两类：

- **节奏型（engagement loop）**：今日 + 最近一月。真正让用户回来的循环。
- **总结/里程碑型（readout）**：一年 / 十年 / 一生。**不是 grind**，而是"日/月选择累积出来的走向 + 少量大命题"，长尺度从"做题"变成"照镜子"。

## 推荐机制：运限周期键 + 自定节奏 + 跨期自动解锁/归档

1. **每个 scope 有 `periodKey`（当期流曜标识）**，由出生盘 + 当前日期算出：
   - 今日 = 流日干支 + 日期（如 2026-06-19 癸亥）
   - 月 = 当前流月（丙午年甲午月）
   - 年 = 当前流年（丙午）
   - 十年 = 当前大运（乙巳）
   - 一生 = 固定常量
2. **后端按 `(userId, scope, periodKey)` 存一"章"**：该章的卡（或生成种子）、用户选择、完成度、lifeState 快照、时间戳。
3. **进入时比对 periodKey**：
   - 同期 → 续作（接着上次，不重置——即把"月度切 tab 从头重做"那个前端 bug 的修法搬到服务端）；
   - 跨期 → 旧章自动归档进历史，按新流曜生成并解锁新章。
4. **刷新靠日历/运限自动推进**，不靠用户等：到下个月新月章自动出现，跨年新年度命题自动解锁。用户**在章内自定节奏**，不强制一次做完。

## 各尺度落法

| 尺度 | 机制 | 频率体感 |
|---|---|---|
| 今日 | 每日一题，跨日刷新 | 天天回来 |
| 最近一月 | 一章一把卡，几次刷完；跨月解锁新章 | 几天刷完，月度续新 |
| 最近一年 | 少量年度大命题 + 累计走向总结，跨年解锁 | 偶尔看，不 grind |
| 最近十年 / 一生 | 几乎纯总结/镜子：汇总日/月选择，显示长期轨迹 | 随时反映现状，不"做" |

长尺度的"累计走向总结"——**解读页的「综合解读」就是雏形**（reading-view-model.js buildOverall）。后端阶段把它做成按 periodKey 累计 + 跨期对比即可。

## 还要一起想的设计点

1. **lifeState 跨期延续**：不能无限累积撞 0/100。建议跨期向基线（50）部分回归——既延续又留波动空间。
2. **缺勤/断点**：用户隔几个月回来，别堆积补做压力，默认跳到当前期，过往章归档可选回看。
3. **历史叙事**：归档的章 = 用户的"人生回放"，是长期留存钩子（"看看去年的我怎么选的"）。

## 后端待办清单（本阶段不做，进入后端阶段一并处理）

1. **合并为单一题库 + 关卡挂 7 主线（大重构）**：把今日池 `data/guanqia.json` 与游戏尺度池 `data/life-game-templates.json` 合并成单一题库，按 `level`(event/arc) 或 `scopes` 标签区分粒度而非来源；并让每道情境题挂到对应主线主题（career/wealth/relationship/health/mindset/family/network），使"今日情境题 = 某条主线在当下的具体切面"，打通关卡层与解读层。
   - 为什么排后端：① 属后端规则较大重写（动 life-game.js 选卡/打分引擎），撞当前纯前端阶段约束；② 与本文的 periodKey + 按章持久化强耦合，题库的组织/keying/标签应跟存储模型一起定，先做易返工；③ 当前无用户痛点（#7/#8 已清掉数值/文案债），属架构整洁度。
   - 轻量替代（若想在前端阶段先要"关卡↔解读"连接，可不动引擎/不合并文件）：仅给卡加"所属主线"标识，在题目与解读间互相引用。
2. **持久化与刷新机制**：按本文「运限周期键 + 自定节奏 + 跨期解锁/归档」实现（见上文）。
3. **lifeState 跨期延续策略**：跨期向基线（50）部分回归，避免无限累积撞 0/100。
4. **缺勤/断点处理 + 历史归档回看**：见上文设计点。
5. **先天×后天对照（命盘 × 选择）的粒度（待决策，依赖打卡数据持久化）**：未来要做「命盘倾向 vs 你实际怎么选」的对照。结论方向已基本谈拢，等后端有了每日打卡/选择的持久化记录再定具体参数：
   - **不做单次对比**：一道题是噪声不是倾向，用一次选择去对照"一辈子的盘"会过度解读、撞文案红线（一次 ≠ 倾向）、且量级不对（稳的先天 vs 一瞬动作）。
   - **分两层**：① 即时反馈（每答一题）只讲这次选择的朝向/代价（lifeState delta），**不碰命盘**；② 趋势对照（累加到一定量）才引入命盘——pattern-based 表达（"反复/更偏向…，而盘的底色偏稳——你在突破惯性"），趋势 vs 趋势量级对得上，文案安全。先天×后天的"王牌"在第二层发生。
   - **冷启动门槛**：累加对照需设最低条数门槛（暂拟某尺度 ≥3~5 次选择才出对照），未达门槛只显示命盘倾向本身、不硬比。
   - **待定**：触发节点（按尺度攒满 / 按主题宫位攒满——后者接「命盘页某宫挂一句你最近的选择」）、门槛具体条数。
   - 为什么排后端：现在前端 `gameSession.choices` 只在内存、刷新即丢，没有跨期/每日打卡记录可累加，对照无数据可依。
