# 后端阶段设计备忘：多尺度关卡的持久化与刷新机制

> 这是**后端改造阶段**才落地的设计，当前纯前端阶段不做（见 CLAUDE.md 阶段约束）。
> 记录于 2026-06-19 的一次讨论：纯前端版选择只在内存、刷新即丢；做后端要记录用户、且
> 今日/月/年/十年/一生这几种关卡不可能"一年做一次"，需要一套刷新/持久化机制。

## 核心判断：五个尺度不能用同一种"刷 N 题"机制

五个尺度的自然频率差几个数量级（日/月/年/十年/一生）。不能都做成"play through N 张卡"。分两类：

- **节奏型（engagement loop）**：今日 + 最近一月。真正让用户回来的循环。
- **总结/里程碑型（readout）**：一年 / 十年 / 一生。**不是 grind**，而是"日/月选择累积出来的走向 + 少量大命题"，长尺度从"做题"变成"照镜子"。

## 推荐机制：运限周期键 + 自定节奏 + 跨期自动解锁/归档

1. **每个 scope 有 `periodKey`（当期流曜标识）**，由出生盘 + 当前日期算出：
   - 今日 = 流日干支 + 日期（如 2026-06-19 癸亥）
   - 月 = 当前流月（丙午年甲午月）
   - 年 = 当前流年（丙午）
   - 十年 = 当前大运（乙巳）
   - 一生 = 固定常量
2. **后端按 `(userId, scope, periodKey)` 存一"章"**：该章的卡（或生成种子）、用户选择、完成度、lifeState 快照、时间戳。
3. **进入时比对 periodKey**：
   - 同期 → 续作（接着上次，不重置——即把"月度切 tab 从头重做"那个前端 bug 的修法搬到服务端）；
   - 跨期 → 旧章自动归档进历史，按新流曜生成并解锁新章。
4. **刷新靠日历/运限自动推进**，不靠用户等：到下个月新月章自动出现，跨年新年度命题自动解锁。用户**在章内自定节奏**，不强制一次做完。

## 各尺度落法

| 尺度 | 机制 | 频率体感 |
|---|---|---|
| 今日 | 每日一题，跨日刷新 | 天天回来 |
| 最近一月 | 一章一把卡，几次刷完；跨月解锁新章 | 几天刷完，月度续新 |
| 最近一年 | 少量年度大命题 + 累计走向总结，跨年解锁 | 偶尔看，不 grind |
| 最近十年 / 一生 | 几乎纯总结/镜子：汇总日/月选择，显示长期轨迹 | 随时反映现状，不"做" |

长尺度的"累计走向总结"——**解读页的「综合解读」就是雏形**（reading-view-model.js buildOverall）。后端阶段把它做成按 periodKey 累计 + 跨期对比即可。

## 还要一起想的设计点

1. **lifeState 跨期延续**：不能无限累积撞 0/100。建议跨期向基线（50）部分回归——既延续又留波动空间。
2. **缺勤/断点**：用户隔几个月回来，别堆积补做压力，默认跳到当前期，过往章归档可选回看。
3. **历史叙事**：归档的章 = 用户的"人生回放"，是长期留存钩子（"看看去年的我怎么选的"）。

## 与现状衔接

当前前端 `gameSession.{choices, lifeState, activeScope}` 已是这套模型的内存版。迁移缺的就是：**periodKey 维度 + 持久化 + 跨期解锁/归档**。玩法骨架不用推翻。相关前端逻辑：`src/app/store.js`（recordChoice 去重、按 scope 存选择）、`src/domain/view-models/game-view-model.js`（按已答卡续作 + 完成态）、`reading-view-model.js`（综合解读）。
