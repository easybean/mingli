# Mingli Product UX Refactor Plan

本文件用于约束 Mingli 重构流程。后续所有前端改动必须先对齐本文件，再进入代码实现。

## 0. 执行规则

### 固定流程

1. Product Manager：确定本轮目标、边界、顺序、验收标准。
2. Product Designer：确定用户路径、页面职责、内容优先级。
3. Mobile UX：先设计手机端，再适配桌面端。
4. Fortune Game Designer：设计游戏循环、情绪钩子、继续动机。
5. UI Reviewer：实现后检查视觉层级、间距、按钮、可读性。
6. Frontend Coder：只有前面方案确认后才允许写代码。

### 禁止事项

- 禁止先写代码再补设计理由。
- 禁止把后端数据结构原样摊到页面。
- 禁止在首页首屏展示书籍、知识命中、检索片段、原始依据。
- 禁止手机端只是压缩 PC 版布局。
- 禁止一页同时承载总览、解读、游戏、数据依据四种任务。

## 1. Product Manager 总控

### 本轮目标

把 Mingli 从“命理资料展示页”重构为“用户能立即开始体验的人生选择游戏 + 深度解读工具”。

### 为什么现在做

当前页面内容已经足够多，但用户进来后不知道先做什么。最大问题不是信息不足，而是：

- 没有主线。
- 首页像摘要仓库。
- 人生游戏还像解读内容的另一种展示。
- 手机端缺少真正的产品化重排。

### 当前不做

- 不继续扩书。
- 不继续扩命理规则。
- 不新增复杂账号、存档、付费系统。
- 不做 AI 生成内容。
- 不做全站视觉大换血，先重构关键路径。

### 阶段顺序

1. 首页/总览：建立用户主路径。
2. 人生游戏：从列表展示改成逐题推进。
3. 今日关卡：重做成高频、短平快、可复测体验。
4. 近期关卡：最近一月、最近一年、最近十年分层。
5. 深度解读：变成解释层，而不是默认入口。
6. 命盘/时间轴：作为工具层，服务深入用户。

### 每阶段完成标准

- 用户 3 秒内知道下一步。
- 手机端首屏只服务一个主动作。
- 每页只有一个主路径，辅助内容不抢焦点。
- 页面文案能被非命理用户理解。
- 游戏选择有情境、张力、反馈和下一步。

## 2. Product Designer 方案

### 用户目标

用户打开 Mingli 时，核心不是“看一堆命理资料”，而是想知道：

1. 我现在处在什么人生阶段？
2. 我今天/近期该注意什么？
3. 我的长期主线是什么？
4. 我该怎么选择，才能把局面走好？
5. 如果我想深究，依据在哪里？

### 推荐主路径

```text
出生信息
-> 生成命盘
-> 今日关卡
-> 选择路线
-> 获得反馈
-> 继续最近一月 / 人生主线
-> 需要时进入深度解读
```

### 首页职责

首页不是解读页，也不是数据页。首页只做三件事：

1. 让用户输入或确认出生信息。
2. 告诉用户当前最推荐看的内容。
3. 把用户分流到今日、近期、人生主线、深度解读。

### 首页首屏结构

手机端首屏建议：

1. 标题区：一句话定位产品，例如“用命盘开启今天的人生关卡”。
2. 出生信息输入区：保持紧凑，不做大表单感。
3. 主按钮：生成 / 重新生成。
4. 生成后主卡：今日关卡入口。
5. 次入口：最近一月、人生主线。

不该出现在首页首屏：

- 十二宫完整信息。
- 大段八字/紫微解释。
- 书籍来源。
- 知识命中。
- 检索片段。
- AI Prompt Seed。

### 页面职责表

| 页面 | 用户问题 | 页面职责 | 不承担 |
| --- | --- | --- | --- |
| 首页/总览 | 我现在该点哪里？ | 路径分发、当前重点 | 展示全部解释 |
| 人生游戏 | 我会遇到什么选择？ | 逐题推进、选择反馈 | 一次列完全部题 |
| 今日关卡 | 今天怎么过？ | 高频短任务、复测入口 | 长篇命理解读 |
| 近期关卡 | 近期怎么安排？ | 最近一月/一年/十年节奏 | 具体灾难预测 |
| 深度解读 | 为什么这么说？ | 解释命盘结构和依据 | 首屏默认入口 |
| 命盘 | 我的盘是什么？ | 结构化查看盘面 | 游戏主路径 |
| 时间轴 | 运势如何展开？ | 阶段变化和大运流年 | 今日操作建议 |

## 3. Mobile UX 方案

### 手机端原则

- 先手机，后桌面。
- 单列为主。
- 一个屏幕只强调一个主动作。
- 卡片要短，不能把 PC 卡片直接缩小。
- 导航标签必须一行展示，不依赖横向滚动提示。

### 手机首页结构

```text
[出生信息/生成按钮]
[今日关卡主卡]
[最近一月入口] [人生主线入口]
[身份摘要 2-4 个短标签]
[深度解读入口]
```

### 手机人生游戏结构

```text
[当前关卡进度]
[情境标题]
[短剧情]
[三个选择横向或紧凑卡片]
[选择后反馈]
[下一题按钮]
[行动日志折叠]
```

### 手机阅读密度

- 普通正文 line-height 控制在 1.5-1.65。
- 游戏题目正文 line-height 控制在 1.45-1.55。
- 卡片 padding 移动端控制在 14-18px。
- 同类信息不要用大段落重复解释。
- 每张卡优先 3-5 行内表达清楚。

### 手机导航

建议保留 5 个一级标签，但必须短：

- 总览
- 游戏
- 解读
- 命盘
- 时间

如果仍然拥挤，优先把“命盘/时间”收进“更多”，不要让导航换行。

## 4. Fortune Game Designer 方案

### 用户为什么会打开

- 想知道今天有什么提醒。
- 想确认近期选择是否合适。
- 想用轻松方式理解自己的命盘。
- 想获得一种“这是关于我”的反馈。

### 核心情绪

第一版不要追求复杂，而要让用户产生：

- 被点中。
- 有选择。
- 想继续。
- 愿意明天再来。

### 游戏循环

```text
今日钩子
-> 当前情境
-> 三个选择
-> 即时反馈
-> 属性变化
-> 下一题/近期路线
-> 结果总结
```

### 今日关卡要求

今日关卡必须比人生主线更具体、更生活化。

不合格示例：

- “事业压力测试”
- “关系边界课题”
- “资源整合阶段”

合格方向：

- “今天有人临时把麻烦推给你，你要不要接？”
- “你发现一个能表现自己的机会，但会打乱原计划。”
- “一段关系里，对方又在试探你的底线。”

### 选择设计

每题固定 3 个选择：

1. 主动路线：更快推进，代价是波动上升。
2. 稳健路线：降低风险，收益更慢。
3. 修复路线：先处理关系、身体或长期结构。

每个选择必须有：

- 现实动作。
- 代价。
- 收益。
- 属性变化。
- 反馈文案。
- 下一题入口。

### 留存点

第一版可先做轻量留存，不做账号：

- 今日结果摘要。
- 当前路线名称。
- 连续完成天数预留字段。
- 分享文案预留字段。
- 明日提示位。

## 5. UI Reviewer 验收标准

### 最影响观感的问题

实现后必须重点检查：

- 页面是否所有卡片看起来同等重要。
- 首屏是否有明确主按钮。
- 是否仍像后台表单。
- 是否有过大的空白和过长的行间距。
- 是否使用太多边框卡片造成碎片感。

### 会影响使用的问题

- 手机导航是否换行。
- 选择按钮是否还是纵向散落且缺少对比。
- 今日关卡是否一眼能读懂。
- 选择后是否知道下一步。
- 长内容是否默认占满屏幕。

### 可继续打磨的问题

- 色彩是否过于单调。
- 卡片标题是否有足够区分度。
- 按钮是否有主次。
- 今日、近期、人生主线是否有不同视觉语气。
- 深度解读是否降级为辅助入口。

### 每次实现后的审查输出

每次页面修改后，必须输出：

1. 最影响观感的问题。
2. 会影响使用的问题。
3. 可继续打磨的问题。
4. 建议先修哪两个。

## 6. 第一阶段待确认方案：首页/总览

### 本阶段目标

把首页从“生成后展示很多摘要”改为“生成后引导用户开始体验”。

### 用户路径

```text
输入出生信息
-> 点击生成
-> 页面自动聚焦今日入口
-> 用户选择：今日关卡 / 最近一月 / 人生主线
-> 如需解释，再进入深度解读
```

### 首页内容优先级

1. 今日关卡主入口。
2. 最近一月入口。
3. 人生主线入口。
4. 身份摘要。
5. 深度解读入口。

### 首页不展示

- 完整大运列表。
- 完整十二宫。
- 章节目录。
- 书籍来源。
- 知识命中。
- 检索片段。
- 专题依据。

### 首页验收

- 手机端首屏能看到生成动作或今日入口。
- 生成后用户不用找，能直接看到下一步。
- 首页不再像资料拼盘。
- CTA 只有一个主入口和两个次入口。
- 深度解读是辅助入口，不抢今日关卡焦点。

## 7. 第二阶段待确认方案：人生游戏

### 本阶段目标

把人生游戏从“全部事件列表”改为“逐题推进的选择游戏”。

### 游戏页结构

1. 当前路线和进度。
2. 当前题目。
3. 三个选择。
4. 选择反馈。
5. 下一题按钮。
6. 行动日志折叠。
7. 阶段关卡折叠。

### 验收

- 页面上默认只出现当前一题。
- 三个选择在桌面端一排展示，手机端可紧凑堆叠。
- 每次选择后下一题根据当前选择更新，而不是锁死第一条路线。
- 行动日志不重复“上一手”。
- 属性超过 100 后仍可表达成长，例如改成等级、突破或溢出显示。

## 8. 第三阶段待确认方案：今日关卡

### 本阶段目标

把今日关卡做成最有留存价值的入口。

### 今日关卡结构

1. 今日一句话钩子。
2. 今天最可能出现的生活情境。
3. 三个可执行选择。
4. 选择后的即时反馈。
5. 今日路线总结。
6. 明日再来提示。

### 验收

- 不出现晦涩命理词作为主标题。
- 每题都像真实生活情境。
- 读完 10 秒内能做选择。
- 选择后有反馈，而不是只显示属性变化。
- 有“明天再来”的理由。

## 9. 后续确认流程

每次进入一个阶段前，先确认：

1. 本阶段是否就是当前最重要问题。
2. 是否需要调整页面职责。
3. 是否要先画文字版结构再实现。
4. 验收标准是否清楚。

确认后才进入代码实现。
